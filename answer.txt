- Composite key cho bảng trung gian
+ Nếu juction table sử dụng khoá chính bằng cách kết hợp 2 khoá ngoại đến từ 2 bảng được (table A, table B) join:
	+ Phải sử dụng 1 object (chứa FkA và FkB) để làm ID.
		+ Object này thực tế là 1 class chứa 2 field là 2 Fk đến từ 2 bảng cần join
			+ Class này phải được đánh dấu là @Embeddable (cho phép nhúng)
			+ Tên các field và name của @Column có thể đặt tuỳ thích (tuy nhiên khi map sang class Entity trung gian thì cần phải lưu ý)
	+ Class Entity cho table trung gian
		+ Sử dụng @EmbeddedId cho field được khai báo với dataType là Object làm Id bên trên
			+ Khi khai báo như thế, JPA sẽ hiểu rằng: Id của table này sẽ là sự kết hợp từ id.fieldA + id.fieldB (truy cập field thông qua object id)
		+ Sử dụng @MapId
			+ Giá trị trong @MapId("###") sẽ phải giống hệt với field trong Object được sử dụng làm Id tương ứng
				+ JPA sẽ hiểu rằng: id.fieldA sẽ là giá trị của biến mà đang được sử dụng @MapId("fieldA")
				tức là: x.getId() sẽ được gán giá trị cho biến id.fieldA
		+ Sử dụng @JoinColumn
			+ Giá trị trong @JoinColumn("###") sẽ là tên cột mà field được khai báo sử dụng để ghi giá trị 
			+ Nên để giá trị ### này giống với giá trị trong @Column mà Object được khai báo vì chúng đều sử dụng chung 1 giá trị
			+ Nếu không có @JoinColumn thì JPA sẽ tự động sinh ra tên cho cột cho field được khai báo bên bảng trung gian này sử dụng
			có thể nó giống với tên cột tương ứng mà bên Object đang được sử dụng hoặc tên khác => rủi ro không đồng bộ

- Instant vs LocalDateTime
	+ Instant: chỉ 1 thời điểm xác định xảy ra hành động, nó không quan tâm tới UTC, nó lưu ngày giờ của chuẩn hệ thống (UTC) và có thể convert để hiển thị phù hợp với các UTC khác nhau.
		   Nó sẽ lưu thời điểm được tính theo UTC chuẩn (tức là UTC 0), ví dụ nếu đang ở UTC +7 thì Instant sẽ lưu thời gian trừ đi 7 tiếng để về với UTC 0
	+ LocalDateTime: bị phụ thuộc vào timezone tức là nó sẽ lưu thời gian đúng như hiện tại nhưng nhược điểm là sẽ không thể biết thời gian này đang lưu ở vùng UTC nào

- Generic Type: <T>
	+ T là viết tắt của Type (đương nhiên là có thể đổi bằng bất kì chữ nào)
	+ chưa biết kiểu dữ liệu, sẽ quyết định sau
	+ nếu không dùng <T> thì có thể dùng Object để thay thế, tuy nhiên khi dùng Object sẽ không an toàn bằng dùng Generic Type
	  bởi vì: khi dùng Object, có thể dữ liệu trả về không đúng với loại mong muốn nhưng chương trình không đưa ra cảnh báo nào
	  còn khi dùng <T>, lúc trả về dữ liệu, chúng ta đã phải chỉ ra rằng phải trả về loại dữ liệu nào, nếu không khớp -> CẢNH BÁO SỚM

- Generic method: 
	+ khi method là STATIC muốn sử dụng Generic Type thì bắt buộc phải khai báo. Ví dụ: public static <T> Response<T> success(T data){}
		+ <T> ở vị trí đầu tiên mang ý nghĩa giống như việc khai báo biến để có thể sử dụng phía sau
		+ <T> ở vị trí thứ 2 và 3 là những nơi mà <T> được sử dụng
	+ method không phải static thì không cần khai báo <T> vì lúc này nó sẽ sử dụng <T> của class

        * bản chất của sự khác nhau trên là do <T> khi khai báo trên class là biến thuộc về đối tượng
	  tức là khi tạo ra 1 đối tượng (object) thuộc về class đó, thì đối tượng đó đã biết được kiểu dữ liệu của mình là gì 
	  nên khi dùng các method (non-static) thì nó sẽ không cần phải khai báo lại kiểu dữ liệu cần dùng là gì
	* trong khi method static có thể chạy mà không cần khai báo object trước đó, vậy nên sẽ cần phải khai báo kiểu dữ liệu cho các method static này

	! thực tế, khi 1 object được khai báo, nó sẽ có thể gọi tới cả method non-static và static, bởi vì khi gọi method non-static thông qua object, 
	  thì Java hiểu ngầm là gọi thông qua class nên vẫn có thể hoạt động được (mặc dù IDE sẽ hiển thị warning)

	* non-static: thuộc về đối tượng
	* static: thuộc về class

- Khi custom Exception (extends RuntimeException):
	+ việc gán message bằng cách super(message) thực chất sẽ trải qua các giai đoạn
		+ gọi tới constructor có tham số (message hoặc tham số tương ứng) ở class cha, ở đây là RuntimeException vì AppException extends nó. Class cha sẽ tiếp nhận message 
		+ từ class RuntimeException tiếp tục gọi tới constructor tương ứng của class cha là Exception để truyền message
		+ từ class Exception tiếp tục gọi tới constructor tương ứng của class cha là Throwable để truyền message
			+ trong constructor này, message sẽ được gán cho biến detailMessage
	+ từ đó mới có thể lấy ra được message khi đứng từ object AppException thông qua method: getMessage()
		+ class AppException không có method này nhưng nó extends cha là RuntimeException nên nó sẽ tìm trong class cha
		+ trong class RuntimeException cũng không có method này nhưng nó lại extends cha là Exception nên nó sẽ tìm trong class cha
		+ trong class Exception cũng không có method này nhưng nó lại extends cha là Throwable nên nó sẽ tìm trong class cha
		+ trong class Throwable có method getMessage(), mà method này lại trả về detailMessage được gán ở bước trên nên nó có thể lấy ra chính xác message thông báo lỗi

- OOP: coi mọi thứ là các thực thể, đối tượng. Những đối tượng này có các thuộc tính (data) và hành vi (method).
	+ Encapsulation (tính đóng gói): Giấu dữ liệu bên trong 1 lớp vỏ bọc, chỉ cho phép truy cập thông qua getter, setter. Mục đích là để bảo vệ tính toàn vẹn của dữ liệu
		+ Vậy nên: sẽ phải để access modifier cho các field là private để tránh việc có thể set trục tiếp giá trị cho biến mà không thông qua setter
		+ Việc dùng setter có mục đích chính là để check logic, validate trước khi set giá trị cho field
		+ Tại sao không dùng public kết hợp với việc check logic trước khi set giá trị cho field, việc này tương tự với check logic trong setter mà?
			+ Làm như thế thì vẫn được nhưng không an toàn. Ví dụ, trong 1 team, bạn có thể nhớ và check giá trị trước khi set nhưng người khác thì có thể là không nhớ mà vẫn cứ set giá trị cho biến -> lỗi.
			+ Việc dùng setter để check logic sẽ giúp dev có thể an tâm khi set giá trị cho biến mà không cần phải lo về logic của biến đó, đồng thời giúp tiết kiệm thời gian khi tập trung logic lại 1 nơi (trong getter) thay vì logic rải rác khắp các nơi trong code.
	
	+ Inheritance (kế thừa): cho phép 1 class kế thừa thuộc tính và hành vi từ 1 class khác. Mục đích là để tái sử dụng code và mở rộng hành vi.	
		+ Có 2 kiểu kế thừa:
			+ extends: kế thừa class: is - a (class này là 1 phần của class khác): kế thừa logic, state từ abstract class 
				+ 1 class không thể kế thừa nhiều class khác(tức là chỉ được extends tối đa 1 class khác)
				+ cho phép class con sử dụng các field được khai báo ở class cha (trong khi class con không cần khai báo) và có thêm các field khác
			+ implements: implement interface (class này có thể làm những gì của class khác): kế thừa hành vi
				+ 1 class có thể implements nhiều interface khác nhau 
		+ extends cho phép kế thừa cả field (state) và method, trong khi implements interface chỉ cho kế thừa method rồi tự triển khai. Vậy tại sao không dùng luôn extends để thay thế cho implements?
			+ Vì: extends không cho phép 1 class có thể extends nhiều class khác trong khi implements thì cho phép implements nhiều interface cùng lúc. -> khả năng mở rộng về sau

	+ Polymorphism (đa hình): cùng method call nhưng có thể có nhiều hành vi khác nhau, tuỳ thuộc vào đối tượng sử dụng.
		+ 2 loại:
			+ overloading: (nạp chồng): chạy ở thời điểm compile-time. Lí do để cho overloading chạy ở thời điểm này đó là quá trình compile sẽ chỉ dựa vào datatype của parameter, method name. Vậy khi compile, nó đã xác định được kiểu dữ liệu của biến, tên hàm cần gọi tới vì vậy nó đã biết được tên hàm và kiểu dữ liệu của các biến trong hàm cần gọi là nhưng không thể biết chắc chắn method trong class cha hay class con được gọi tới, vì chúng ta có thể new 1 đối tượng con thông qua cha, mà quá trình compile sẽ chưa thể xử lí được phần tạo đối tượng, vậy nên nó sẽ chỉ check được biến được tạo ra thuộc class nào, class đó có method đang được gọi tới không, nếu không thì compile fail, nếu có thì tiếpp tục compile. 
			
			+ overrding: (ghi đè): chạy ở thời điểm runtime. Quá trình runtime sẽ diễn ra việc gọi constructor, tạo đối tượng. Vậy nên đến lúc này, mới có thể xác định được chính xác method mà object call tới, vì đã các định được chính xác đối tượng thuộc class nào.

- Abstract class và Interface: cả 2 đều dùng để thể hiện tính trìu tượng
	+ abstract class: là 1 loại class đặc biệt, giúp định nghĩa LÀ GÌ, tuy nhiên:
		+ KHÔNG THỂ new ra 1 đối tượng bằng cách nào đi chăng nữa
		+ có thể chứa abstract method (method không có body)
		+ vẫn có thể khai báo các biến như class thông thường 
		+ đương nhiên là vẫn có thể có các getter, setter cho các biến -> và các method getter, setter này không phải là abstract method. Vậy nên trong abstract class cha vẫn có thể chưa các method bình thường mà không phải là abstract method
		+ nếu 1 class (thông thường) extends abstract class thì nó sẽ phải implement và triển khai mọi abstract class của cha để triển khai
		+ 1 abstract class con extends abstract class cha thì nó có thể implements và triển khai luôn 1 phần các abstract method của cha, còn nếu không triển khai thì abstract con sẽ vẫn ngầm định implement các abstract method của class cha
		+ việc class con triển khai code từ các method có ở cha thì sẽ không bắt buộc phải có @Override, mặc dù chúng cùng tên, nhưng nếu không có @Override thì sẽ không được coi là Overriding method cha. Tuy nhiên vẫn nên dùng @Override để Java có thể check được việc mismatch tên hàm.
		+ 1 abstract class vẫn có thể extends 1 class (thông thường) khác, và nó vẫn giữ nguyên các đặc điểm của 1 class con (có thể sử dụng lại các state, method của cha) và mở rộng thêm các method khác 
		+ không thể new ra 1 object con rồi object đó gọi tới 1 method chỉ có trong class của con mà không có trong class cha. Vì khi compile-time chạy, nó chưa tạo Object với đối tượng con, mà nó đang coi việc tạo object con thông qua cha (Parent child = new Child()) chỉ dừng tới việc Parent child, tức là child sẽ là Object của Parent nên sẽ không thể call method mà không có trong class Parent được. 
		+ tại sao cần phải là abstract class mà không phải là class thường? 1 class vẫn có thể extends 1 class thường mà?
			+ Vì: nếu là class thường thì mọi thứ sẽ không được hợp lí. Khi là class thường, ta có thể tạo đối tượng cho class đó, tuy nhiên đối tượng này gần như vô nghĩa, không có tác dụng gì.
			+ Khi là class thường, method phải có body, tuy nhiên, trong body này chứa logic thì không hợp lí, mà đúng ra các logic sẽ phải do từng method trong từng object kế thừa tự xử lí logic cho riêng nó.

			
	+ interface: KHÔNG phải class, nó là 1 kiểu riêng biệt, có thể coi như 1 bản thiết kế hành vi thể hiện khả năng làm được gì.
		+ không thể tạo field bên trong interface, không có constructor
		+ access modifier mặc định luôn là public và là abstract method // những thứ này có sẵn từ khi Java xuất hiện -> những thứ này chẳng khác gì bên abstract method ở abstract class
			+ Lí do mà access modifier là public vì nó phải được phép gọi tới từ bên ngoài 1 class khác khi các object implements nó được khởi tạo và cần dùng tới các method.
		+ từ Java 8 trở đi, ngoài abstract ra, còn xuất hiện thêm keyword: default. Với default method, chúng ta có thể tạo body cho các method ngay tại interface, không bị giới hạn như các abstract method của các phiên bản trước đây (không cho phép method có body). Từ đó dẫn tới việc các class implements interface có thể dùng luôn các method này hoặc override nó để phù hợp với class của riêng nó.

- Compile-time & Runtime
	+ Compile-time: (thời điểm biên dịch): (chạy lệnh javac để biên dịch file .java -> .class)
	+ Mục đích:
		+ Kiểm tra syntax
		+ Kiểm tra type
		+ Kiểm tra method có tồn tại không
		-> phát hiện lỗi sớm, đảm bảo type safety, giúp code an toàn trước khi chạy

	+ Runtime: (thời điểm chạy): (chạy lệnh java)
	+ Mục đích: 
		+ Load class
		+ Gọi constructor
		+ Tạo OBJECT
		+ Call method
		+ Quản lí memory
		-> thực thi logic, xử lí đa hình, xử lí tương tác thực tế.

	+ Array: 
		+ 1 vùng nhớ liên tục -> cho phép CPU truy cập 1 phần tử trong Array rất nhanh, O(1) vì, địa chỉ = base + index x size
		+ điểm yếu: size cố định, không mở rộng thêm được. Lí do để nó là fixed size là: array được lưu trữ trong memory liên tục, vậy nên nếu resize thì phải tìm block nào đó đủ lớn -> khó
	+ ArrayList
		+ bản chất cũng là Array, nhưng có thể linh hoạt thay đổi kích cỡ của mảng
		+ Vấn đề là nếu ArrayList có bản chất là Array thì nó thay đổi kích cỡ của mảng bằng cách nào?
			+ Nó thay đổi size bằng cách copy toàn bộ các phần tử ở array cũ sang 1 array mới có kích cỡ gấp khoảng 1.5 lần mảng cũ
		+ Độ phức tạp
			+ get index để lấy element: O(1) (vì nó giống Array)
			+ add:
				+ nếu còn chỗ trống (size < capacity) -> O(1)
				+ nếu không còn chỗ trống -> copy sang mảng mới rồi mới thêm được -> O(n)

			+ remove(index): 
				+ nếu xoá ở cuổi -> nhảy tới thẳng index đó rồi xoá -> O(n)
				+ xoá ở chỗ bất kì -> xoá xong rồi phải dịch chuyển lại mảng để không cho phép null ở vị trí phần tử vừa bị xoá

	+ LinkedList: bản chất là double linked list: tức là mỗi phần tử (node) sẽ liên kết tới phần tử ngay phía trước và phía sau nó
		+ Độ phức tạp:
			+ truy cập phần tử: O(n): vì cần phải duyệt từng node để tới node cần lấy
			+ add, remove:
				+ đầu, cuối: O(1), vì chỉ cần node.prev hoặc node.next nên rất nhanh 
				+ giữa: O(n), mặc dù thêm thì chỉ mất O(1) nhưng thời gian để tới được chỗ cần thêm/xoá thì là O(n)

	+ List: là 1 interface mà cả ArrayList và LinkedList đều implements nó
